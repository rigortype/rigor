## Open Engineering Items

Persistent items that have come up across v0.0.x slices and that the next implementer benefits from seeing without re-reading the full thread. Items already absorbed into v0.1.1 are referenced through MILESTONES rather than restated here.

1. ~~**C-body classifier indirect mutators.**~~ Closed in v0.1.2 — the extractor's seed now also matches `_modify` / `_modifiable`-named helpers whose body issues a frozen-check (`str_modifiable`, `rb_struct_modify`, `range_modify`, `rb_class_modify_check`, …). Catalog regen flips `String#replace` / `String#initialize_copy` / several String bang methods / `Range#initialize` / `Range#initialize_copy` to `mutates_self`. A transitive-closure-on-first-arg-formal-param approach was explored and reverted — over-classification on functions where the first arg is a formal yet the helper doesn't actually mutate it. Per-class blocklists still absorb the remaining cases (Time / Set helpers).
2. ~~**RBS::Extended grammar lacks Symbol / String literal tokens.**~~ **Closed.** `ImportedRefinements::Parser` now recognises `:name` and `"name"` inside a type-arg position, plus `T1 | T2` unions of those literals. Three new AST nodes (`TypeNode::SymbolLiteral` / `StringLiteral` / `Union`) flow through the resolver pass as `Type::Constant` / `Type::Union` carriers. `Pick[T, :a | :b]` / `pick_of[Shape, "a" | "b"]` now round-trip end-to-end through `ImportedRefinements.parse`.
3. **Sig-gen `update_existing` does not yet collapse sibling parent / child class blocks.** Gap (c)'s tree-builder fix lives in `Writer#render_new_file` (the create-new path). When updating an existing target file, `merge_class` still resolves each candidate's `class_name` independently — if both `Foo::Bar` and `Foo::Bar::Child` decls already exist as flat siblings, sig-gen leaves them flat. Re-flowing an existing file into the nested layout would require parsing the existing decl tree and rewriting it, which is out of scope for a follow-up fix. Users who want the canonical nested layout regenerate from scratch (delete the target sig file and rerun).
3. ~~**ADR-14 follow-up: kind tracking + `module_function` + nested-class emission.**~~ **All five sub-items now closed.** The third-round `lib/rigor/analysis/` dogfood surfaced four interacting gaps the generator did not handle without hand-edits; (a) / (b) landed in `b6d3286`; (c) / (d) / (e) landed this session.
   - ~~**(a) module-vs-class detection on intermediate segments.**~~ **Closed by commit `b6d3286`.** `Generator#walk_defs` records every walked `ClassNode` / `ModuleNode` into `@namespace_kinds[qualified_name] => :class | :module`, threads through `MethodCandidate#namespace_kinds`, and the writer emits per-segment kinds with `module` as the safe default.
   - ~~**(b) Leaf-kind detection for `module Foo; module_function`.**~~ **Closed by commit `b6d3286`.** `walk_defs` tracks `module_function` (no-arg) directives via a special-cased `walk_statements` so the flag carries across sibling statements; `method_def_prefix` emits `def self?.name` for marked defs.
   - ~~**(c) Nested Data-define classes inside another class.**~~ **Closed this session.** `Writer#render_new_file` now builds a namespace tree keyed by qualified-name segments instead of `group_by(&:class_name)`. Every candidate whose `class_name` is a strict prefix of another is nested inside the parent's block; e.g. `Rigor::Analysis::DependencySourceInference::GemResolver::Resolved`'s methods now appear inside the same `module GemResolver` block that carries GemResolver's own `resolve` / `locate_gem_spec`. The same tree mechanism consumes per-candidate `class_shells` from gap-(e).

5. ~~**`Method#curry` precision through `Type::BoundMethod`.**~~ **Closed (Option A).** `Inference::MethodDispatcher::MethodFolding.try_backward` now recognises `:curry` alongside `:call` / `:[]` and treats it as identity on the carrier — `<bound>.curry` keeps the same `(receiver_type, method_name)` pair, so a subsequent `<curried>.call` still routes through the recursive dispatch the backward fold already wires. `[:to_i, :to_f, :to_sym].map { |m| "1".method(m).curry.call }` now infers to `Tuple[Constant<1>, Constant<1.0>, Constant<:"1">]` instead of `Tuple[Dynamic[Top]×3]`. Partially-applied forms (`.curry(n).call(a)`) still lose precision and fall through to `Type::Combinator.untyped` — the faithful `Type::CurriedBoundMethod(receiver_type, method_name, accumulated_args)` carrier (Option B) is reserved for a future slice when concrete user demand surfaces.
6. **In-memory `Analysis::Runner.run_source` entry point (test-only perf follow-up).** The `RunnerHelpers#analyze` test helper materialises a tmpdir per call (write source file, chdir, run, clean up). At ~25-50ms per call × hundreds of runner-spec calls, that's a real share of suite wall-clock that an in-memory entry point could eliminate. Sketch: add `Runner.run_source(source:, path: "code.rb", environment:, config:)` that bypasses path expansion and accepts a `{path => bytes}` virtual file table. The helper would call it for the `analyze(source: "...")` shape (no files / sig). Expected delta: ~5% sequential, ~3% parallel — not worth doing standalone, but a natural complement if Track 7 work expands the test suite further.
7. **Fork-based file-level parallelism for `rigor check` (single-shot CLI).** Stackprof of warm `rigor check lib` shows ~50% inference, ~22% `Marshal.load`, ~17% GC. Ruby's GVL serialises CPU-bound work, so a Thread-based file-walk parallelism investigation (prototyped and reverted; commit history) was NEGATIVE — wall-clock grew from 1.85s to 2.15s at `RIGOR_WORKERS=4` because thread coordination overhead exceeded the (zero) GVL-released gains. The path that would actually help: **fork worker processes** that each analyse a slice of files. Cost: ~50-100ms per fork on macOS + per-process `Environment` rebuild (the in-memory blobs would need either pre-fork warm-up + COW sharing or post-fork cache-hit). Net benefit only for projects with hundreds of files or a multi-second analysis tail. Implementation sketch: `Runner#run` shells to `parallel` gem or `Process.fork` directly, splits `expansion.fetch(:files)` into worker chunks, each worker writes its diagnostics array to a pipe, parent re-assembles in original path order. Cache::Store is already thread-safe (Monitor around `@memo` + counters) so the per-process Store inside each fork is safe even if forks share the underlying directory.
8. **Ractor migration — bound by [ADR-15](adr/15-ractor-concurrency.md).** Phases 1 + 2a + 2b + 3a + 4a + 4b + 4c landed. Phase 1: every `Rigor::Type::*` / `Rigor::TypeNode::*` / `Cache::Descriptor` / `FactStore` / `FlowContribution` is `Ractor.shareable?` at construction. Phase 2a: `Rigor::Configuration` is `Ractor.shareable?`. Phase 2b: new `Rigor::Environment::Reflection` value object extracts the loader's read-only RBS query surface; it is `frozen?` but NOT `Ractor.shareable?` because the cached `RBS::Definition` objects transitively reference C-extension `RBS::Location`. The Phase 4 worker pool sidesteps this by having each worker build its OWN Reflection from the shared `Cache::Store`. Phase 3a: new `Rigor::Plugin::Blueprint` value object is frozen + `Ractor.shareable?` and carries `klass_name` (String — the constant path) plus a deep-frozen `config` Hash; `Plugin::Registry` now exposes `blueprints` (aligned `Array<Blueprint>`) and gains `Registry.materialize(blueprints:, services:)` for per-Ractor replay. Plugin INSTANCES intentionally stay non-shareable — `rigor-sorbet`'s `@reachable_absurd_nodes` / `@reveal_type_calls` / `@assert_type_mismatches` accumulators and every Rails-plugin `*_index` Hash are the canonical reasons. The Phase 4 pattern is ship blueprints across the Ractor boundary, materialise once per worker, own instances locally. Phase 4a: new `Rigor::Analysis::WorkerSession` value carrier — per-worker analysis substrate consuming only `Ractor.shareable?` inputs (`Configuration`, `cache_store`, `Array<Plugin::Blueprint>`, `explain`) and building its OWN `Plugin::Services` + materialised `Plugin::Registry` + `DependencySourceInference::Index` + `Environment` + per-session `RbsExtended::Reporter` + `BoundaryCrossReporter` internally. Plugin `#prepare` runs once at construction; raises captured into `#prepare_diagnostics`. `#analyze(path)` is the per-file substrate (equivalent of `Runner#analyze_file` + `plugin_emitted_diagnostics` + `explain_diagnostics`); `#drain_reporters` returns frozen reporter snapshots for end-of-pool merge. Severity-profile re-stamping intentionally left to caller. Equivalence with `Runner#analyze_file` proven by [`spec/rigor/analysis/worker_session_spec.rb`](../spec/rigor/analysis/worker_session_spec.rb). Phase 4b: `Rigor::Analysis::Runner#initialize` gains a `workers: N` keyword (default `0` = sequential, the documented v0.1.4-and-earlier path). When `N > 0`, the new private `#analyze_files_in_pool(files)` spawns N Ractors; each takes the shareable payload (Configuration, cache_root String, `Array<Plugin::Blueprint>`, explain) and builds its own WorkerSession inside the Ractor body. Workers write back to the main Ractor via `Ractor.main.send` (Ruby 4.0+ mailbox model — `Ractor.yield` was removed). Three message kinds: `[:prepare, diagnostics]` (coordinator keeps the FIRST snapshot only — `#prepare` is deterministic per plugin so N workers' identical snapshots dedupe to one), `[:file, path, diagnostics]` (one per file), `[:done, drained_reporters]` (one per worker at exit). Diagnostic order is preserved by re-flowing the per-path result Hash through the original input order before flattening. Per-worker reporter snapshots merge via `record_*` APIs (dedup naturally). `Environment::ClassRegistry.default` made `Ractor.make_shareable` so workers can lazy-read it without `Ractor::IsolationError`; the runner pre-warms it on the main Ractor before pool spawn. WorkerSession constructor no longer writes the `MethodDispatcher::FileFolding.fold_platform_specific_paths` process-global (non-main writes raise) — the Runner sets it once on the main Ractor. Equivalence + plugin replay + prepare-dedup proven by [`spec/rigor/analysis/runner_pool_spec.rb`](../spec/rigor/analysis/runner_pool_spec.rb). Phase 4c: three opt-in surfaces (precedence CLI > env > config > 0) — `rigor check --workers=N`, `RIGOR_RACTOR_WORKERS`, `.rigor.yml` `parallel.workers:`. `Configuration#parallel_workers` reads the YAML (non-negative Integer; raises on typos); CLI's `resolve_workers` walks the chain; negative env-var values clamp to 0. Sequential remains the documented default. Audit-spec coverage in [`spec/rigor/ractor_readiness_spec.rb`](../spec/rigor/ractor_readiness_spec.rb) (Phases 1, 2, 3, 4a, 4b). **Known limitation — Phase 4b.x deferred:** pool mode workers only reliably handle TRIVIAL source files; non-trivial files trip `Ractor::IsolationError` on first `RBS::EnvironmentLoader.new` access (chain of non-shareable RubyGems / RBS module constants). Pool spec excluded from default `make verify` via `spec_helper.rb`'s `exclude_pattern`; run via `RIGOR_INCLUDE_RACTOR_POOL=1 bundle exec rspec spec/rigor/analysis/runner_pool_spec.rb` or `make test-ractor-pool`. **Next: Phase 4b.x — worker-side env-build stability.** Three candidate fixes outlined in [`docs/design/20260514-ractor-migration.md`](design/20260514-ractor-migration.md) § "Phase 4b.x" (coordinator-side env + cross-Ractor handle, RubyGems / RBS upstream patches, or pivoting to fork-based parallelism). Until then, default sequential is the production path. ADR-15 is the binding contract; ADR-4 (Environment) / ADR-6 (Cache::Store) / ADR-2 (Extension API) carry matching boundary notes.

## Where the Work Resumes

1. **O4 Layer 3 slice 3 (per-gem-version cache descriptor) — DEFERRED until measured pain.** The user-facing surface of slice 3 (graceful-degradation `:info` diagnostic for uncovered gems) has landed; the remaining piece is per-gem-version cache invalidation: tracking `(gem_name, gem_version)` in `Cache::Descriptor::RbsCollectionEntry` so a `bundle update` of one gem invalidates only that gem's slice, not the whole project's RBS env cache. ### Stdlib coverage expansion — open design questions (2026-05-16)

Driven by the v0.1.5 work to expand `Environment::DEFAULT_LIBRARIES` (commits `0a4ffea` + follow-on). Twenty-three additional stdlib libraries now auto-load (1,273 → 1,412 RBS classes). Two design judgment points surfaced and are deferred:

1. **Stdlib name conflicts with rigor's internal carriers (RESOLVED).** `singleton` (stdlib `Singleton` mixin) initially clashed with `Rigor::Type::Singleton` in lexical scope, surfacing an `undefined-method` false positive on `lib/rigor/type/singleton.rb#==`'s `is_a?(Singleton)` narrowing. Resolved via the lexical-nesting fix in `Inference::Narrowing#analyse_class_predicate`: the bare-name argument now resolves through the chain derived from `scope.self_type` (mirroring `Module.nesting`-driven constant lookup) so the nested `Rigor::Type::Singleton` wins over the top-level stdlib `Singleton`. `singleton` is now in `DEFAULT_LIBRARIES`. Broader lexical-resolution coverage (method-dispatch receivers, attribute access, etc.) is gated on further demand — the fix is surgical to the class-predicate narrowing surface.

2. **Stdlib RBS coverage gaps in upstream `rbs` gem.** `strscan` initially blocked Batch 2 because the bundled RBS only declares `StringScanner#[](Integer)`, while real Ruby supports `StringScanner#[](Symbol)` for named captures — which rigor itself uses in `lib/rigor/builtins/imported_refinements.rb:422,424`. The expedient resolution (response path (a) below) applied two `# rigor:disable argument-type-mismatch` directives at the affected call sites and moved `strscan` into `DEFAULT_LIBRARIES`, so users analysing parser code get StringScanner dispatch precision. The remaining response paths stay open as longer-term fixes:
   - **(a) Skip the library** — initially used in Batch 2; superseded by Batch 3.5 in favour of (a')
   - **(a')** **In-source `# rigor:disable` directives** at the gap-affected call sites + load the library. What landed in Batch 3.5. Trade-off: rigor's own code carries a maintenance burden when upstream RBS gets fixed (must remove the directives); the burden is bounded (two call sites in one file). This is the same pattern already used for `Bundler::LazySpecification#platform` in `lockfile_resolver.rb`.
   - **(b) Hand-author an RBS overlay** under rigor's own `sig/`. Per AGENTS.md § "RBS Authorship" the project policy is to prefer `rigor sig-gen` over hand-authored RBS, but "filling an upstream RBS shim gap" is arguably a different category from "generating from inference". A small focused overlay (`StringScanner#[](Symbol)` row) would let rigor's own code drop the disable directives.
   - **(c) Upstream the fix to `ruby/rbs`** and bump the gem. Highest-quality long-term path; needs cross-project coordination.

   The same pattern will recur for other stdlib RBS gaps as the library set expands. The decision tree is: when an upstream RBS gap is surfaced by a single internal call site, prefer (a') (disable directive); when it surfaces across multiple call sites or in user-facing code, escalate to (b) or (c).

### Pre-survey persistent items

1. **Sig-gen `update_existing` does not yet collapse sibling parent / child class blocks.** Gap (c)'s tree-builder fix lives in `Writer#render_new_file` (the create-new path). When updating an existing target file, `merge_class` still resolves each candidate's `class_name` independently — if both `Foo::Bar` and `Foo::Bar::Child` decls already exist as flat siblings, sig-gen leaves them flat. Re-flowing an existing file into the nested layout would require parsing the existing decl tree and rewriting it, which is out of scope for a follow-up fix. Users who want the canonical nested layout regenerate from scratch (delete the target sig file and rerun).
2. **In-memory `Analysis::Runner.run_source` entry point (test-only perf follow-up).** The `RunnerHelpers#analyze` test helper materialises a tmpdir per call (write source file, chdir, run, clean up). At ~25-50ms per call × hundreds of runner-spec calls, that's a real share of suite wall-clock that an in-memory entry point could eliminate. Sketch: add `Runner.run_source(source:, path: "code.rb", environment:, config:)` that bypasses path expansion and accepts a `{path => bytes}` virtual file table. The helper would call it for the `analyze(source: "...")` shape (no files / sig). Expected delta: ~5% sequential, ~3% parallel — not worth doing standalone, but a natural complement if test-suite expansion continues.
3. **Fork-based file-level parallelism for `rigor check`.** Stackprof of warm `rigor check lib` shows ~50% inference, ~22% `Marshal.load`, ~17% GC. The Phase 4b Ractor path handles the parallelism story for v0.1.5; a fork-based path remains a parallel (non-exclusive) option for hosts where Ractors are unavailable or where COW sharing of pre-warmed `Environment` blobs would beat per-Ractor env build. Implementation sketch: `Runner#run` forks workers per file-chunk, each writes diagnostics to a pipe, parent re-assembles in original path order.
